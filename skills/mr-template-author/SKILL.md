# MR template author

You are helping the user create or modify a Merge Request description template for a project managed by **MCC Taganrog** (a macOS CI/CD server with a Slack bot for Flutter teams). MR descriptions are auto-generated by the bot when developers run `cc <task>` in Slack — the bot creates a worktree, makes changes, commits, pushes, and creates a Draft MR in GitLab using a configurable template.

## Skill scope

Use this skill when the user wants to:
- Author a new MR description template for a specific project or use case
- Modify an existing template (add fields, restructure, change style)
- Validate that a template uses correct placeholder names
- Decide which placeholders are right for a given audience (junior vs senior reviewers, internal vs external teams, etc.)

Do **not** use this skill for:
- Generic GitLab/GitHub MR template questions unrelated to MCC
- Editing other types of templates (commit messages, Slack notifications, system prompts)
- Implementation work in the bot itself (rendering pipeline, loader, schema) — those belong to MCC repo work, not user-facing templates

## How MR templates work in MCC

1. Templates are written in **Mustache** (`{{var}}`, `{{#section}}...{{/section}}`, `{{^inverted}}...{{/inverted}}`).
2. The bot's render pipeline: build context → load template by priority chain → Mustache render (no HTML escape, markdown mode) → review-text sanitize → whitespace normalize → secret-blob gate → publish to GitLab.
3. Per-project override: `~/mcc/<project>/templates/mr/<name>.md`, activated via `~/mcc/<project>/config.yaml > git.mr_template: <name>`.
4. Bot default: `~/mcc/slack-bot/templates/mr/<name>.md`, activated via `~/mcc/slack-bot/config.yaml > mr_active_template: <name>`.

For full architecture details, the user can run `mcc mr-template list` (Slack) or read the MCC docs.

## What a good MR template does

- **Scannable in 10 seconds** — reviewer can answer "what is this, who made it, should I care?" without scrolling.
- **Progressive disclosure** — long content (AI review output) goes inside `<details>` collapsibles.
- **Conditional sections** — review section only appears if review ran; retry marker only if it's a retry; target-override warning only if explicit override was set.
- **No empty sections** — if the data is null, the whole section disappears (use Mustache truthy / inverted blocks).
- **No secrets, no internal paths, no costs in dollars** — these are explicitly stripped or never included by design (the render pipeline has guards, but a good template doesn't reach for them).

## Workflow when the user asks you to author a template

1. **Read the variable reference** — see `reference/variables.md`. Pick placeholders that match the audience and use case the user described.
2. **Read the Mustache cheatsheet** — see `reference/mustache-cheatsheet.md`. Pay special attention to the gotchas (e.g., `changed_files_list` requires `has_changed_files` guard).
3. **Look at examples** — `reference/examples/default.md` is the canonical full template; `reference/examples/minimal.md` is a compact alternative. Use them as scaffolds.
4. **Generate the template** as a single markdown file. The user typically saves it as `~/mcc/<project>/templates/mr/<name>.md` and activates with `mcc mr-template set <name>` in Slack.
5. **Validate by mental walkthrough**: imagine these scenarios — does each render correctly without empty sections, broken markdown, or duplicated blocks?
   - Success + review OK
   - Success + `cc noreview` (no review section)
   - Review failed (timeout / spawn error / cancelled by user)
   - Retry of a previous task
   - Target override (`cc target=develop`)
   - Multi-commit MR (the trick `{{#commit_messages_all.1}}` shows the list only when ≥2 commits)
   - No `changed_files_list` available
6. **Tell the user how to install and test it**:
   - `mcc mr-template add <file.md>` (if you provide a downloadable file) OR copy the file manually into `~/mcc/<project>/templates/mr/`
   - `mcc mr-template preview <name>` in Slack — renders on mock data so the user can see the result before activating
   - `mcc mr-template set <name>` to activate
   - First MR after activation will use the new template

## What NOT to put in a template

These guards exist in the render pipeline, but a good author doesn't write code that reaches for them:

- `{{worktree_path_absolute}}` — reveals server filesystem path. Use `{{worktree_path_relative}}` if you need a path at all (rare).
- `{{coder_cost_usd}}` is intentionally absent from the schema — on Max subscription the figure misleads. Show tokens (`{{coder_tokens_in}}` / `{{coder_tokens_out}}`) if you need a "cost" indicator.
- Full CLI model id — show `{{model_short}}` (`sonnet`/`opus`), not `claude-sonnet-4-6`.
- HTML `<script>`, `<iframe>` — markdown sanitizer escapes them, but they have no place in MR descriptions anyway.
- External tracking pixels in review text — sanitizer strips them, but don't try to embed them.

## Common pitfalls

1. **`changed_files_list` double render** — use `{{#has_changed_files}}` as the outer guard, not `{{#changed_files_list}}`. The latter iterates as array and re-renders the inner section per file.
2. **Multi-line `prompt` in blockquote** — `prompt` is auto-preprocessed: every `\n` becomes `\n> `. So `> {{prompt}}` works correctly without manual handling.
3. **Conditional `{{commit_messages_all.1}}`** — this is the Mustache trick for "if length ≥ 2": access second element. Combined with `{{#commit_count}}` outer guard, it shows the multi-commit list only when actually multi-commit.
4. **`<details>` nesting** — the renderer auto-escapes `</details>` and `</summary>` inside `review_text_full`, so wrapping review text in your own `<details>` is safe.
5. **Empty fallback for nullable URLs** — `{{commit_url}}`, `{{slack_thread_link}}`, `{{user_name}}` can be null. Wrap in `{{#commit_url}}[link]({{commit_url}}){{/commit_url}}` to skip the section gracefully.
6. **Validation** — after authoring, the user should run `mcc mr-template preview <name>` in Slack. The bot's `validateTemplate` will warn about unknown placeholder names, but won't catch logic errors like the double-render trap.

## When in doubt

- For broad audience (external contributors, mixed-skill team): use a structure close to `reference/examples/default.md`. Tables, checklists, full review block.
- For internal team that ships fast: use a structure close to `reference/examples/minimal.md`. One-line metrics, no checklist, footer with task id only.
- For specific compliance needs (Jira links, mandatory fields, security review marker): add those as plain text or markdown headers — there's no special placeholder for them, just write what's needed in markdown.
